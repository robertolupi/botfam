# Doc Metadata

1. Use frontmatter for documentation metadata
   1. Pro: easy to parse for bots
   2. Pro: easy to read in Obsidian and can be indexed with Obsidian bases
2. Define canonical properties for different kinds of documents
   1. Pro: have proper linter that checks, enables doc refactoring

## Canonical Properties

Frontmatter is YAML between `---` fences as the first bytes of the file. All
YAML lists/arrays must use block-style notation (e.g. `  - item` on its own
line) for Obsidian native compatibility. Properties the linter would check
(advisory first, see Transition Plan):

### Common to every doc kind

| Property  | Required | Values                                                                     |
| --------- | -------- | -------------------------------------------------------------------------- |
| `kind`    | yes      | `proposal` \| `review` \| `session` \| `design` \| `protocol` \| `lineage` |
| `status`  | yes      | see per-kind vocabularies below                                            |
| `authors` | yes      | list of actor names (`rlupi`, `claude`, `agy`, …) or external model ids    |
| `created` | yes      | `YYYY-MM-DD`                                                               |

### Per kind

- **proposal** — `status`:
  `Draft | Proposed | Approved | Implemented | Rejected | Tabled | Superseded`
  (same taxonomy as [doc/template/proposal.md](../template/proposal.md)).
  Additional: `proposal-id`, `executor`, `quorum`, `deadline` — identical to
  the `!propose` line so tooling can cross-check frontmatter against the CCREP
  ledger. `superseded-by`: path or proposal-id, required when
  `status: Superseded`.
- **review** — `status`: `Final` (retrospectives are point-in-time records and
  never reopen). Additional: `session-date`, `actor`.
- **session** — generated by `botfam irclog2sessions`; the generator owns the
  frontmatter (`status: Generated`, `channel`, `started`, `ended`). Hand edits
  are a lint error.
- **design** — `status`: `Live | Historical`. `Historical` docs keep their
  bodies frozen (banners-not-rewrites rule from the 2026-06-11 doc audit); the
  linter flags body edits to `Historical` docs.
- **protocol** — `status`: `Live`. Only PROTOCOL.md and its satellites; always
  current truth.
- **lineage** — `status: Historical`, frozen bodies, same rule as design.

### Relationship to existing status banners

The human-visible `> [!NOTE]` status banners stay — they render on GitHub where
frontmatter does not. Frontmatter is the machine copy of the same fact; the
linter checks the two agree rather than replacing one with the other.

## Linter Integration

The frontmatter checks land in the doc-linter
([doc/proposals/doc-linter.md](doc-linter.md)) as a new check class:
required-properties-per-kind, value vocabularies, banner/frontmatter agreement,
frozen-body enforcement for `Historical` docs, and CCREP cross-checks for
proposals. agy implements this in Go (volunteered 2026-06-12 on #botfam).

## Transition Plan

1. **Step 0 (this change):** add `mdformat-frontmatter==2.1.2` to
   `tools/mdformat.sh` so the canonical formatter preserves YAML blocks.
   Verified against the existing corpus: zero reflow. Without this, a plain
   `tools/mdformat.sh` run mangles any frontmatter in doc/.
2. **Step 1:** ratify this schema via CCREP; frontmatter this file (done) as
   the worked example.
3. **Step 2:** backfill frontmatter across doc/ mechanically (one commit per
   kind, like the 85c41d8 normalization — big but semantically empty diffs).
4. **Step 3:** doc-linter gains the frontmatter checks, advisory mode.
5. **Step 4:** flip to blocking once the corpus is clean — same convention as
   mdformat.sh: run-before-commit, enforced socially + by review, since git
   hooks are not versioned.
