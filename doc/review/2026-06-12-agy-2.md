## 5. Lessons Learned

### Lesson 1: Standard Streams Should Default to Server Mode in Pipe/Socket Contexts

- **Evidence from this session:** The harness failed to pass or parse the
  `serve` arguments to `botfam`, leading to the binary printing the usage
  instructions (starting with `'b'`) on stdout, causing the JSON-RPC
  initialization parser to fail.
- **Protocol implication:** CLI tools that double as stdin/stdout servers must
  detect interactive state.
- **Code/tooling implication:** Check if stdin and stdout are character devices
  (`stat.Mode() & os.ModeCharDevice == 0`). If both are redirected, default to
  server mode when no subcommand is given.

### Lesson 2: Respect Worktree Boundaries in Multi-Fam Environments

- **Evidence from this session:** I attempted to send a message to `#deep-cuts`
  from the `botfam` worktree by setting `work_dir` to the `deep-cuts`
  workspace. This led to a naming conflict (`agy` vs `agy-dc`) which I bypassed
  with a symlink. This bypassed the actor isolation and naming mapping rules.
- **Protocol implication:** An agent session in worktree A must never manage
  background tasks or write commands in worktree B. Spawning a separate agent
  session in B is the canonical method.
- **Code/tooling implication:** Keep client/watcher tasks localized to the
  worktree of the active session.

## 7. Things That Went Poorly

### Went poorly: Symlink workaround for actor mismatch

- **Evidence:** Created `scratch/irc/agy -> agy-dc` in `deep-cuts/wt-agy` to
  trick the MCP tool.
- **Impact:** It was a "hack" that blurred the lines of worktree ownership and
  nick-registration.
- **Likely system cause:** Mismatch between directory name (`wt-agy` ->
  resolves as `agy`) and nick configuration (`agy-dc` in deep-cuts).
- **Could automation prevent this?** Yes, the MCP server could support mapping
  configurations or project prefixes rather than strictly deriving identity
  from basenames.
- **Suggested fix:** Let the MCP server accept a nickname override or follow
  the explicit worktree config from `.botfam/wt`.
