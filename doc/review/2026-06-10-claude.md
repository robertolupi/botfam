## 1. Correctness bugs

### High

- **H1 — Actor mis-resolution for any hyphenated directory name.**
  `internal/fam/root.go:34` parses the actor as everything after the *last*
  `-` in the workdir basename, with no `wt-`/`botfam-` prefix requirement.
  `deep-cuts` → actor `cuts`; `botfam-foo-bar` → `bar` (docs say `foo-bar`);
  a hyphenated main checkout silently gets a wrong sticky identity. Every doc
  (CLAUDE.md, PROTOCOL.md §1, bootstrap heredoc) documents prefix-strip
  semantics. Only `wt-bob` is tested (`root_test.go:58-94`). Codex flagged
  this in `doc/collab/sessions/bootstrap-botfam-setup/session.md:49`.
  **Fix:** only parse when the basename starts with a known prefix; strip the
  prefix instead of splitting on the last dash; fail closed otherwise.

- **H2 — Sweep vs Heartbeat/Complete race can give a task two owners.**
  `Heartbeat` (`internal/store/store.go:482-490`), `Complete` (`:460-480`),
  and `Abandon` (`:492-515`) do `findClaimed` → `writeJSON(path)` with no
  ownership re-check, and `writeJSON` is truncate-write `os.WriteFile` with
  `O_CREATE` (`:642-649`). A concurrent `Sweep` (`:517-558`, triggered
  implicitly every 15s by `Recv`, and by `Post`/`Claim`) can rename
  `claimed/<actor>/f` → `open/f` between the read and the write; the write
  then *recreates* the file under `claimed/` — the task exists in both
  `open/` and `claimed/` and a second actor can claim it. This is exactly the
  contention window of heartbeat-near-expiry. **Fix:** make in-place updates
  ownership-proving (rename to a private tmp name — fails if swept — write,
  rename back; or `O_WRONLY` without `O_CREATE`, treating ENOENT as "lease
  lost").

- **H3 — merge-gate reviewer spoofing defeats the independent-approval check.**
  `internal/fam/merge_gate.go:181` reads `reviewer` from the message
  *payload*, only falling back to the authenticated `msg.From`
  (`:233-236`). The store binds sender identity at the MCP boundary, but the
  gate trusts the payload over it: an author can send
  `ccrep:evaluation {reviewer: "bob", verdict: "approve"}` from their own
  mailbox and pass the gate guarding `main`. **Fix:** payload `reviewer` must
  never override `msg.From` / `entry.Actor` (or must match it). The test
  suite does not cover this case.

### Medium

- **M1 — `SessionRead` silently truncates sessions.**
  `internal/store/session.go:155-173`: default 64KB `bufio.Scanner` limit and
  `scanner.Err()` never checked. One long markdown entry (no size cap at
  append time) silently drops itself and every later entry — including from
  the promoted `session.md` via `SessionRender`. **Fix:** enlarge the scanner
  buffer to a documented cap, check `scanner.Err()`, enforce the cap in
  `SessionAppend`. (The new `TestSessionTornLineTolerant` covers corrupt
  JSON, not this.)

- **M2 — Single-threaded stdio loop: a blocking `recv` stalls everything.**
  `internal/mcp/server.go:393-420` handles requests strictly sequentially
  with `context.Background()`, so a default 120s `recv` blocks pings,
  cancellation, and — critically — `heartbeat` for a claimed task from the
  same session. **Fix:** dispatch requests on goroutines (the currently
  unused `writeMu` at `:395` already exists to serialize writes) and honor
  `notifications/cancelled`. Bears on KNOWN_ISSUES §9 and §18.

- **M3 — Non-atomic task updates → torn reads.**
  All in-place task writes (Ack `store.go:252`, claim stamp `:374`/`:451`,
  Complete `:469`, Heartbeat `:489`, Abandon `:504`, Sweep `:545`) use
  truncate-write while other processes concurrently read the same files. A
  torn read in `Sweep` is silently skipped (`:537`), so an expired task can
  be missed; in claim-by-id status checks it yields wrong answers.
  **Fix:** route updates through the existing `writeJSONAtomic` (`:610-622`),
  currently used only for creates.

- **M4 — Malformed `expires_at` → instant silent expiry.**
  `internal/mcp/server.go:517-523`: `argFloatPtr` coerces a present-but-non-
  numeric `expires_at` (e.g. a string) to `0`, and `expire` treats 0 as
  already past — message loss with no error. **Fix:** error on present-but-
  non-numeric values.

- **M5 — Bound actor vs `work_dir` mismatch unchecked after binding.**
  `internal/mcp/server.go:357-364`: `callActor`/`envActor` vs `dirActor`
  conflicts are now rejected (improved by `4222756`, integration-tested),
  but an already-bound `s.actor` is never checked against a later call's
  `work_dir` — after binding as alice, a call with `work_dir` resolving to
  bob proceeds silently as alice. **Fix:** also reject
  `s.actor != "" && dirActor != "" && s.actor != dirActor`.

- **M6 — merge-gate fails open when the proposal event is missing.**
  `internal/fam/merge_gate.go:84-98,130`: no `ccrep:proposal` found →
  `author=""` (any approval counts as independent) and `latestCommitSHA`
  falls back to the requested commit (supersession check skipped). A
  `ccrep:revision` with a missing/empty `commit_sha` likewise
  un-supersedes. **Fix:** fail closed when the proposal event is absent or a
  revision lacks `commit_sha`.

- **M7 — `session close` failure modes.**
  `internal/fam/session_cli.go:162-191`: (a) clean-tree is checked *before*
  close renders and stages `session.md`, so an aborted `git commit -e`
  leaves the file staged and a rerun fails its own clean-tree check —
  wedged until manual cleanup or `BOTFAM_FORCE_CLOSE=1`; (b) close never
  writes the `ARCHIVED` tombstone (`session.go:266-279`), so a closed
  session stays in `SessionList` and still accepts `session_append` — only
  a test writes the tombstone by hand (`session_test.go:106-108`).

### Low

- **L2** — `Resolver.getenv` falls back to the real process env even when an

## 4. API/design smells

- `claim` no-match returns `null` but claim-by-id no-match is an error
  (`store.go:457` vs `:418`) — two shapes for "didn't get a task".
- `Task` has no `created_by`; `post` validates then discards the actor while
  abandon/sweep provenance is kept.
- `send` to an unknown actor silently creates a dead mailbox
  (`store.go:133`); the `fam.toml` roster is never consulted.
- Mixed time-unit naming: `timeout_s` vs `lease_ttl` (relative seconds) vs
  `expires_at` (absolute epoch) — invites the M4 class of caller mistakes.
- `session_read` uses `from` where every other tool uses `actor`.
- Empty `handoff: {}` silently treated as no handoff (`server.go:290`),
  bypassing `SessionAppend`'s validation.
- The per-actor flock covers `recv`/`try_recv`/`ack`/`session_append` but
  not `claim`/`complete`/`heartbeat`/`abandon` — the single-process-per-actor
  invariant is enforced unevenly.
- Implicit sweeps from `Post`/`Claim`/`Recv` plus an explicit `sweep` tool
  make `swept_from` attribution hard to reason about — document in the tool
  descriptions.
