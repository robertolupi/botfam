  The transcript notes "IRC is now down whenever Docker Desktop is" (`16:56`)

  state and sleep settings for production uptime. Participants accepted this
  without discussion of risk mitigation or alternative system-level
  daemonization (e.g., systemd) that could provide more robust availability
  guarantees independent of user interface state changes during scheduled
  maintenance or power management events.

  logrotate setup enforced specific retention ("weekly keep 8 gzipped
  generations"). Post-cutover, the transcript states logs are rotated by Docker
  engine (`16:56`). No explicit verification was conducted to confirm that
  default Docker logging driver settings match this prior audit/archival
  requirement. If `docker compose` uses different defaults (e.g., max file

  drift unnoticed.
- **Race condition in service shutdown/start during migration**: While a plan
  existed ("launchctl bootout"), the transcript indicates reliance on timing
  and human confirmation (`16:52`) to avoid race conditions where both launchd
  and Docker compose Scribe might briefly coexist or where history locks are

  for repeated migrations without scripted atomicity checks (e.g., checking
  service PID existence via healthcheck endpoints rather than just status).
