Reading your 2026-06-11 session log, botfam just proved it can shepherd a real multi-agent day — but the day also exposed where the harness is still doing IRC workarounds instead of using its own primitives.

**What landed cleanly**

- Scribe now stamps the Git SHA at build time, announces it on join, and answers `!version`. That's the exact `-ldflags` discipline Claude proposed.
- You moved it from an ad-hoc session to launchd (`com.botfam.scribe, KeepAlive`), then an hour later cut over to Docker without losing history — prod ergo v2.18.0 + scribe now runs as `botfam-irc-prod` with data bind-mounted.
- The hermetic test substrate works: `compose.test.yaml` + `docker/test-substrate.sh` spins ergo and scribe, joins #botfam, asserts `!tally`.
- Log rotation shipped (weekly, keep-8), then naturally migrated to Docker logs when you moved prod.

Those are real wins. The friction points are where botfam's own design — "maildir-backed mailbox with a *blocking* receive" and "state under `~/.botfam/<project>/`, atomic rename, no daemon" — wasn't used yet.

## Pain points the log shows

1. **Stale-binary class, round 2** — version.go fell back to `exec git rev-parse HEAD` in the CWD, so an old `~/bin/botfam` could confidently report the wrong SHA. You fixed it with `debug.ReadBuildInfo()`, but the bug existed because the build didn't embed vcs info reliably.

2. **Flag parsing order** — scribe errored before parsing `--file` because it resolved `COLLAB_HISTORY` first. agy patched it, but it's a classic "fail before flags" pattern.

3. **Double-scribe risk** — Claude had to say "Not starting a second scribe; double-appends to the ledger would be worse than none". You manually coordinated unload/load during Docker cutover.

4. **irc-wait re-arming** — the new Go subcommand replaced shell watches, but Claude noted the sharp edge: "watcher likely was not re-armed after the last response — re-arming after every wake is the sharp edge". Same bug existed for `(hist)` replay lines.

5. **Process leaks in tests** — the substrate was built partly to "fixing the test-server process leaks (~190 reaped today)". Docker helps, but the root is orphaned `botfam serve` processes.

6. **Log path drift** — after migration, "chat.log now lives at ~/botfam-irc/data/chat.log — sessions pipeline input path changed". Your `irclog2sessions.py` broke silently.

## Proposals — make botfam eat its own dogfood

### 1. Kill irc-wait, use botfam recv
You already have a blocking receive. Instead of polling IRC, give each agent a mailbox:
- `botfam recv --wait --filter "irc:#botfam"` blocks until scribe writes a new line to `~/.botfam/botfam/irc/`
- scribe becomes a translator: IRC → maildir, not the coordination primitive

Benefit: no re-arming, no `(hist)` filtering in clients, and Claude/agy stop rebuilding shell loops.

### 2. Make scribe a botfam task worker
Your `!propose /!vote /!tally` workflow is a lease-based queue in disguise. Map it:
- `botfam post --queue proposals` instead of IRC `!propose`
- `botfam claim` gives the reviewer a lease, `complete` records the verdict
- scribe just renders the queue state to IRC for humans

This removes the "double-append" fear — the maildir rename is atomic, and botfam already enforces one claim at a time.

### 3. Enforce singleton with botfam locks
Replace manual launchctl coordination with:

```go
// at scribe startup
lock := botfam.AcquireLock("scribe-singleton", ttl=30s)
defer lock.Heartbeat()
```

If a second scribe starts, `AcquireLock` fails fast. No more "please bootout before cutover."

### 4. Bake version correctly, once
- Drop the git exec fallback completely (you did). Add to Makefile:
  ```
  LDFLAGS=-X main.version=$(shell git describe --always --dirty) -buildvcs=true
  ```
- In CI, fail the build if `debug.ReadBuildInfo().Main.Version == ""`. That guarantees every binary announces truthfully, even ad-hoc builds.

### 5. Docker as the default dev substrate, not an afterthought
- Promote `compose.test.yaml` to `compose.dev.yaml` — developers run `docker compose up` and get ergo + scribe + a shared `COLLAB_ROOT` volume
- Add a healthcheck that runs `botfam inbox --json | jq.` to catch the ~190 process leaks early
- Mount `~/.botfam` read-write, so local `botfam` CLI and containerized scribe share the same maildir (no COLLAB_HISTORY env hacks)

### 6. Centralize paths via fam.toml
Instead of hardcoding `~/botfam-irc/data/chat.log`, write:

```toml
[paths]
chat_log = "${BOTFAM_DATA}/chat.log"
history = "${BOTFAM_DATA}/history.jsonl"
```

botfam already parses minimal fam.toml — extend it. Then `irclog2sessions.py` reads the same config, no more silent breakage after migration.

### 7. Add a real test for the flag bug
Create `internal/fam/scribe_test.go`:
- Test 1: run `scribe --file /tmp/x` outside a fam worktree → should not error on missing COLLAB_HISTORY
- Test 2: run inside worktree without flag → uses default
Table-driven, runs in CI inside the Docker substrate. This would have caught the bug before Claude found it live.

# Claude's review of this review

Author: **claude** (botfam fam member, this worktree)
Date: 2026-06-11

**Read the section above as a good auditor working from an outdated briefing.**
The diagnosis is sharp; the central prescription contradicts a decision the
fam made — with operator approval — the same day. Verified against the code
at the time of recording (`agent/claude` @ 6efc1e8).

## What is accurate

The six pain points all match what actually happened in the 2026-06-11
session, with correct quotes: stale binary SHA, flag-before-parse failure,
double-scribe risk, irc-wait re-arming footgun, test process leaks (~190
reaped), and the silent `chat.log` path drift that broke
`irclog2sessions.py`. The "what landed cleanly" list is also correct.

## What is stale or wrong — do not be misled

1. **Proposals 1 and 2 relitigate the IRC-first pivot in reverse.**
   "Kill irc-wait, use `botfam recv`" and "scribe as a renderer over the
   task queue" would make the mailbox/queue substrate canonical and IRC a
   thin view. The fam decided the opposite on 2026-06-11 (operator
   ratified): IRC is the primary coordination channel; scribe/tally bots
   replace the SQLite+UDS substrate. Meta.ai did not have that context.
   It also missed a practical driver of the pivot: `recv`/`send` CLI argv
   payloads defeated harness auto-approval, creating per-message friction
   that IRC does not have. The shared principle — "don't run two parallel
   coordination systems" — is correct; the fam applied it by choosing IRC.

2. **It describes a botfam that doesn't exist.** It quotes the design as
   "maildir-backed mailbox … atomic rename, no daemon". In the actual code
   the store is SQLite (`internal/store/sqlite.go`), `recv` goes through a
   daemon request (`internal/fam/collab_cli.go`), and "maildir" survives
   only in one comment. `botfam.AcquireLock` does not exist anywhere — it
   is an API sketch presented as an existing primitive. (To its credit,
   `recv`/`post`/`claim`/`complete`/`inbox` and fam.toml parsing do exist.)

3. **Meta-lesson:** the repo's older design docs still describe the
   maildir/no-daemon era convincingly enough to mislead a fresh reader.
   Worth fixing at the source, not just here.

## What to keep

These converge with the ChatGPT session-lifecycle proposal
(`doc/protocol/session-lifecycle-and-gtd.md`) — two independent reviewers
landing on the same items is a useful signal:

- **#3 scribe singleton guard** — same as ChatGPT §6.1 / AI-3. The
  lock-with-TTL/heartbeat shape is good design input despite the invented
  API.
- **#4 version baking + CI guard** — "fail the build if BuildInfo is
  empty" is a concrete, cheap addition nobody else proposed.
- **#6 centralize paths in fam.toml** — directly addresses the silent
  `irclog2sessions.py` breakage; fam.toml parsing already exists. Probably
  the best original idea in this review.
- **#7 flag-order regression test** — concrete, table-driven, would have
  caught the scribe `--file` bug before it was found live.
- **#5 compose.dev promotion** — reasonable, but drop the "shared
  `~/.botfam` maildir mount" detail (stale model).

## Verdict

Fold #3/#4/#6/#7 (and a trimmed #5) into the action list. **Reject #1 and
#2 explicitly** — the "which substrate is canonical" question was settled
on 2026-06-11 in favor of IRC and should not be silently reopened by a
future reader of this document.
