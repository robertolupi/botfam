## 5. Lessons Learned

### Lesson 1: Never approve a PR from an assumption about its diff — read the actual tip

Evidence from this session:

- On #45 I re-approved twice inferring "head moved only via a base-merge,
  content unchanged"; on the third look (forced by an empty grep) the
  implementation had materially changed and contained a settings-clobber
  data-loss bug. I retracted APPROVED → REQUEST_CHANGES.

Protocol implication:

- A review verdict must be tied to a specific reviewed SHA; when the head
  moves, re-read the diff between the last-reviewed SHA and the new tip before
  re-approving. (Now codified in `skills/forge-autonomy` §2/§4.)

Code/tooling implication:

- Approvals already auto-stale on head move (good); the gap was *my* behavior,
  not the gate. A `pr-reviewer` subagent that always diffs the tip would make
  the discipline mechanical.

### Lesson 2: When the platform already provides a primitive, configure it — don't reimplement it

Evidence:

- The entire `ccrep` engine/ledger/bang-line/merge-gate stack reimplemented
  what Gitea branch protection does natively (Required Approvals,
  dismiss-stale, block-on-rejected, restrict-push, author-exclusion). It was
  deleted (#34) and replaced by config + a linter. Role tiers, too, are native
  (per-branch rules).

Protocol implication:

- Only build custom coordination for what the forge genuinely cannot express
  (e.g. presence-aware quorum) — and even that is YAGNI until needed.

Code/tooling implication:

- Config that *is* the gate needs a linter (`forge-gate.sh check`) so drift is
  caught (it found `main` at 1 approval).

### Lesson 3: An agent's forge token must carry write + notification scopes

Evidence:

- A token re-mint via `forge-login.sh` produced a **read-only** token (the
  default scopes lacked write); I then 403'd on opening PRs and on
  `/notifications`. Fixed by adding
  `write:repository,write:issue,read/write:notification` to the
  `forge-login.sh` default.

Protocol/tooling implication:

- Per-agent onboarding (`forge-setup.sh`/`newfam`) must mint a token with the
  full agent scope set; a read-only agent silently can't
  propose/vote/merge/wake.

### Lesson 4: Forge notifications only reach agents actually running the wake loop — keep an IRC fallback

Evidence:

- A live test (operator-run) confirmed agy responded to a forge review-request
  within 2 min — but only because agy was on the loop. A request to an agent
  not running `forge-wait` (or without notification scope) would be silently
  unseen.

Protocol implication:

- Escalation rule (now `forge-autonomy` §4): request on the forge; if no
  response in ~2 min, ping on IRC. IRC remains the reliable fallback channel.

## 7. Things That Went Poorly

### Went poorly: premature approval on assumption (#45)

Evidence: re-approved #45 twice without re-reading the tip; missed a real
change.

Impact: a data-loss bug (`writeClaudeSettings` dropping unknown settings keys)
briefly carried an approval; corrected before merge.

Likely system cause: no mechanical "diff-the-tip" step in my review; over-trust
of my own "it's just a base-merge" inference.

Could automation prevent this? **yes** — a `pr-reviewer` subagent that always
diffs the tip + builds, returning structured findings, removes the shortcut.

Suggested fix: practice the subagent-review pattern from `forge-autonomy` §3.

### Went poorly: a destructive working-tree command

Evidence: ran `git checkout botfam-next -- .` which overwrote the working tree
mid-task; recovered with `git reset --hard HEAD` (no commits lost).

Impact: minor (working tree only), but it was an unforced, risky command.

Likely system cause: bundling an unnecessary checkout into a gather command.

Could automation prevent this? partial — avoid `git checkout <other> -- .`;
prefer per-file reads.

### Went poorly: #22 merged a stale roadmap

Evidence: PR #22 merged the roadmap with `(new issue)` placeholders before I
linked #25/#26; required a follow-up (#27).

Likely system cause: fast-merge environment + I didn't confirm what actually
merged. Fix: in fast-merge fams, verify merged content, not the local branch.
