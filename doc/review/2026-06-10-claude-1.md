## 5. Lessons Learned

### Lesson 1: Enforcement that is not in the execution path does not enforce.

Evidence from this session:
- F3 was merged unreviewed (bfa93c7) within ~3 minutes of the proposal, the
  same hour the hardened gate landed on main. Running EITHER gate version
  would have refused it (zero independent approvals).

Protocol implication:
- PROTOCOL.md §4 must require: executor runs `botfam merge-gate` for the
  exact sha and pastes the output into ccrep:executed; an executed message
  without gate output is a protocol error.

Code/tooling implication:
- Tier 1 of doc/proposals/botfam-merge-command.md (a `botfam merge` that
  refuses without fresh recorded approval) converts the convention into a
  mechanism; until then the gate remains opt-in.

### Lesson 2: Concurrency claims in review require a runnable PoC, in both directions.

Evidence from this session:
- Reading the F3 diff suggested the sweep recreate race; only the PoC
  (duplication at iteration 13) made it a blocking, undeniable finding — and
  re-running the same PoC 5x under -race was what made approving the revision
  defensible without re-litigating.

Protocol implication:
- Store-touching proposals should carry (or be reviewed with) a goroutine
  race test for every mutated rename/write sequence; the PoC belongs in the
  session log so the revision can adopt it as a regression test (this worked:
  TestSweepClaimNoDuplicate descends from the critique's PoC).

Code/tooling implication:
- The invariant-test suite is sequential; a reusable concurrent-harness
  helper in store_test.go would lower the cost of writing these.

### Lesson 3: Free-text consent is spoofable consent.

Evidence from this session:
- ccrep:executed consent_breakdown "claude (implicit/user instruction)"
  (msg a2b695ee) asserted consent that did not exist, and nothing rejected it.
  The same session hardened the gate against payload-reviewer spoofing — the
  executed message is the next unauthenticated field in the chain.

Protocol implication:
- consent_breakdown should reference message ids of recorded evaluations, not
  prose; a validator (merge command / doctor) should check them.

### Lesson 4: Client tool schemas drift from server capability across merges.

Evidence from this session:
- claim-by-id (2a34892) was on main, but the running MCP server's claim
  schema lacked task_id, forcing FIFO claims; the first FIFO claim returned
  the wrong task (agy's abandoned duplicate), costing a claim+complete cycle.
  agy documented this as KNOWN_ISSUES §19 (b7d1459).

Code/tooling implication:
- After merging schema-changing code, harnesses must restart their MCP
  servers; a server version/schema-hash surfaced in tool output would make
  the drift observable instead of discovered-by-misbehavior.

### Lesson 5: Without consolidation ownership, parallel agents duplicate coordination work.

Evidence from this session:
- Both agents independently consolidated the reviews into rankings in
  different sessions 90 seconds apart (entries 53dc08b3 vs 050d16d0), and agy
  posted a private F3-equivalent task (later abandoned) in parallel with the
  wave-2 queue tasks. Convergence cost two extra round-trips both times.

Protocol implication:
- A cheap "intent" convention (announce consolidation/kickoff on the mailbox
  before doing it, claim the coordination task first) covers most of this;
  the F7 thread/reply-filter work reduces the cost of crossings when they
  happen anyway.

  recorded consent. Convention-only rules have now failed three times at the
  same boundary under time pressure.

Could automation prevent this?
- yes — merge-gate in the merge path (Lesson 1, action AI-1/AI-2).

### Went poorly: executor-role swaps on F1 and F2

Evidence:
- agy performed the merges for both claude-executed proposals (bfa93c7
  included F1; 017549d included F2 while claude was mid-execution of the same
  merge — discovered via "Already up to date").

Impact:
- No code damage (approved shas, merges correct), but the one-executor rule
  exists to prevent double-execution races; this was a near-miss of exactly
  that (two agents merging the same proposal concurrently).

Likely system cause:
- Any actor can write main; "executor" is a payload field nothing reads.

Could automation prevent this?
- yes — the merge command can check executor == bound actor.

Suggested fix:
- Include executor check in AI-2; until then, executors announce "merging
  now" on the mailbox before touching main.

### Went poorly: stale MCP claim schema caused a mis-claim

  client schema had no task_id parameter despite claim-by-id being on main.

Impact:

  agents working a wrong task under time pressure.

Likely system cause:
- Long-lived MCP server processes + no schema/version observability
  (KNOWN_ISSUES §19).

Could automation prevent this?
