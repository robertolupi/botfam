## 5. Lessons Learned

### Lesson 1: Never Re-Approve on Assumption
- **Evidence**: In PR #45, a base-merge occurred, and a quick re-approval almost went in. However, `claude-bot` caught a data-loss bug that was introduced in a prior commit. Later in PR #47, `claude-bot`'s update triggered another stale approval reset, requiring `agy` to pull and review the exact new tip `d85e7c67` before re-approving.
- **Protocol implication**: Stale approvals must be treated as a complete reset. The reviewer must verify the diff between the last-reviewed SHA and the new tip.
- **Code/tooling implication**: Branch protection rules that dismiss stale approvals on push are crucial for maintaining safety.

### Lesson 2: Map-based JSON Merging Over Rigid Structs for File Editing
- **Evidence**: The initial implementation of `writeClaudeSettings` decoded `.claude/settings.json` into a struct that only defined the fields `EnabledMcpjsonServers` and `Permissions`. When marshaled, it clobbered other valid fields (e.g. `env`, `hooks`, `model`) on the main checkout or during re-runs.
- **Protocol implication**: Tools modifying shared or user configs must selectively mutate fields, preserving unknown properties.
- **Code/tooling implication**: Avoid rigid structs for partial file updates. Use `map[string]json.RawMessage` or similar mechanisms.
