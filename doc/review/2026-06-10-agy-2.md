## 5. Lessons Learned

### Lesson 1: Reconnecting client sockets need explicit CloseWrite to bypass OS reference sharing

Evidence from this session:
- When the daemon replaced an active connection by calling `prev.conn.Close()`, the client's `conn.Read` remained blocked in the test suite. This was due to descriptor references or OS-level socket state sharing in the parent/child process tree.
- Asserting `CloseWrite` and calling it forces a TCP/UDS FIN to be sent, unblocking the peer immediately.

Protocol implication:
- Any persistent peer connection protocol that requires replacing active peers must mandate sending a write-shutdown before closing to guarantee peer awareness.

Code/tooling implication:
- In handlers managing hijacked connections, always check for and invoke `CloseWrite()` to cleanly notify the client of socket termination.

---

### Lesson 2: Reusing `exec.Cmd` objects in Go tests is invalid and causes silent failures

Evidence from this session:
- Three integration tests failed because `cmdTally.Output()` was called twice on the same `exec.Cmd` instance.
- Subsequent calls returned errors like `exec: Stdout already set` and empty output, masking actual behaviors.

Code/tooling implication:
- Always initialize a fresh `exec.Command` instance for each execution in automated tests.

## 7. Things That Went Poorly

### Went poorly: Hijack connection closure behavior debugging

Evidence:
- It was non-obvious why calling `conn.Close()` on the server did not trigger `io.EOF` on the client socket read.

Likely system cause:
- File descriptor inheritance by child processes in the test suite prevented the socket reference count from dropping to zero, which keeps the connection alive at the OS level.

Could automation prevent this?
- No, this is standard POSIX socket behavior.

Suggested fix:
- Document socket shutdown practices in `doc/collab/PROTOCOL.md`.
