* **Date:** 2026-06-10
* **Reviewer:** agy
* **Worktree:** `wt-agy`

## 1. Executive Summary

This document presents a comprehensive review of the `botfam` codebase (Go packages in `cmd/` and `internal/`) as of June 10, 2026. 

While the core functionality of maildir messaging, session locking, and task queues is functional, we identified several critical bugs in task-claiming atomicity and session archiving, along with significant gaps between the implemented codebase and the specification (`doc/DESIGN.md` and `doc/DESIGN_sessions.md`). Furthermore, a critique-bypassing vulnerability in the Phase 1 CCREP merge gate slice was discovered.

## 2. Identified Bugs & Robustness Issues

### A. Session Close & Archiving Deficiencies

#### 1. Missing Tombstone (`ARCHIVED` file)
* **Files:** [session_cli.go](../../internal/fam/session_cli.go#L143-L198) and [session.go](../../internal/store/session.go#L265-L279)
* **Description:** According to [DESIGN_sessions.md](../DESIGN_sessions.md#L43-L44), a session is closed/archived by writing a tombstone file named `ARCHIVED` under `$COLLAB_ROOT/sessions/<slug>/ARCHIVED`. `SessionList()` uses the existence of this file to filter out archived sessions. Currently, `st.SessionClose` renders the session log to Markdown and writes it into the repo, but it **never creates the `ARCHIVED` marker** in the store.
* **Impact:** Closed sessions remain in `botfam session list` indefinitely, cluttering the operator's view unless manually deleted or touched in the state directory.
* **Recommendation:** Update `SessionClose` to write the `ARCHIVED` file inside the store directory upon successful rendering and write to the worktree.

#### 2. Git Status Porcelain Deadlock
* **File:** [session_cli.go](../../internal/fam/session_cli.go#L162-L170)
* **Description:** The `session close` subcommand verifies that the git repository is clean by checking the output of `git status --porcelain`. If a prior run of the subcommand successfully wrote/staged the rendered markdown (`session.md`) but the interactive `git commit` step was aborted by the operator, the rendered file remains staged or untracked. Rerunning `session close` will then be rejected because `git status --porcelain` detects the dirty `session.md` file from the previous attempt.
* **Impact:** The operator is deadlocked and must run manual `git reset` commands to recover.
* **Recommendation:** Modify the `git status` check to ignore modifications to the target file path `doc/collab/sessions/<slug>/session.md`.

### B. Task Queue Claiming Robustness

#### 1. Non-Atomic Claims Leading to Orphaned Tasks
* **File:** [store.go](../../internal/store/store.go#L360-L379)
* **Description:** When claiming a task, `Claim()` renames the task file from `tasks/open/` to `tasks/claimed/<actor>/` as its compare-and-swap (CAS) lock. It then updates the JSON payload of the file with `Status = "claimed"`, the owner, and the lease expiration, and calls `writeJSON` (which is non-atomic and truncates the file). If `writeJSON` fails (e.g. disk full, permission issue), the function returns an error, but the file remains in the claimed directory with its old contents (internal `Status = "open"` and `LeaseExpiresAt = nil`).
* **Impact:** Because the file is in the claimed directory, it is hidden from other workers. Because `LeaseExpiresAt` is `nil`, `Sweep()` ignores it and it is never swept back to open. The task becomes permanently orphaned.
* **Recommendation:** Update task-claiming updates to use `writeJSONAtomic` (which writes to `tmp/` and renames). If writing fails, catch the error, rename the file back to `tasks/open/`, and return the error.

### C. MCP Server Resolution Boundaries

#### 1. Non-Identity Tools Restricting Global Commands
* **File:** [server.go](../../internal/mcp/server.go#L183)
* **Description:** The stdio MCP server calls `resolveActor` unconditionally at the entry point of `callTool` for all tools.
* **Impact:** Read-only global commands (like `session_read` or `sweep`) that do not require any actor context are artificially blocked and throw an `identity required` error if called from outside a member worktree without a `COLLAB_ACTOR` environment variable.
* **Recommendation:** Move `resolveActor` into the individual tool case statements where identity is actually required.

## 3. Consensus (CCREP) Process Vulnerabilities

### A. Critique Bypassing (Old Critique Expiration)
* **File:** [merge_gate.go](../../internal/fam/merge_gate.go#L110-L125)
* **Description:** The `merge-gate` command determines consensus for a proposal by scanning only the evaluations/critiques associated with the **exact** `commitSHA` being checked. If Reviewer A logs a blocking `request_changes` critique on `commit1`, and the author publishes a new revision `commit2`, the critique on `commit1` is ignored. If Reviewer B approves `commit2`, the merge gate succeeds, even if Reviewer A's blocking critiques were never addressed.
* **Impact:** Authors can bypass blocking reviews by simply pushing a minor new commit.
* **Recommendation:** Implement a "latest verdict by reviewer across all commits" rule. A reviewer's latest verdict on the proposal (across all commits) determines their state. A `request_changes` critique on *any* commit continues to block a merge check for *all* subsequent commits, unless that same reviewer submits a newer evaluation with an `approve` verdict.

## 4. Missing Features & Specification Gaps

### A. Tier 1 Receive Ergonomics
* **Specification:** [DESIGN.md §5](../DESIGN.md#L226-L233)
* **Description:** The `recv`, `try_recv`, and `peek` tools are specified to accept `match_from` and `match_reply_to` filters. Currently, only `match_type` is implemented.
* **Impact:** concurrent threads between the same actors cross, and agents cannot long-poll for specific conversation threads.

### B. Conversation Reconstruction (`thread` Tool)
* **Specification:** [DESIGN.md §5](../DESIGN.md#L229-L230)
* **Description:** The read-only `thread(id, limit?)` tool is missing from the MCP server registration and from the backing `store` package.
* **Impact:** Agents cannot trace conversation histories using `in_reply_to` headers.

## 5. Next Steps

1. **Session Archiving Fix:** Update `SessionClose` to write the `ARCHIVED` file to the store upon successful close, and relax git status checks in CLI.
2. **Atomic Task Claiming:** Implement self-healing rollbacks in `Claim()` if write operations fail.
3. **CCREP Gate Critique Persistence:** Re-engineer `MergeGateCmd` to track and persist critiques across different commit SHAs of the same proposal.
4. **Ergonomic Messaging:** Add `match_from`, `match_reply_to`, and the `thread` tool.
