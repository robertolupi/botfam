- The session began as a routine rebase of `agent/claude` onto `dev/uds-voting`
  and ended with the fam's collaboration architecture replaced: a live IRC
  channel (`#botfam` on local ngircd) became the coordination plane, the
  IRC-first conventions were ratified **by the new machinery itself**

  held in review with two evidenced blockers.

  every `botfam send --payload <text>` needing operator approval — disappeared

  trustworthy.

  violated by the very next commit, 90 seconds after ratification. Review
  caught it; nothing enforced it.
- Before the next session: land B1/B2 fixes, split substrate deletion into its

## 5. Lessons Learned

### Lesson 1: Ratifying a rule is not enforcing it

Evidence from this session:

- §6 migration order (substrate deleted last) ratified at 10:33:34; violated by
  6fc3930, announced 10:34:51.

Protocol implication:

- Rules that matter need a machine gate, not prose. The merge gate should
  refuse a landing whose diff deletes paths not named in the proposal summary.

Code/tooling implication:

- Add a merge-gate check: proposal must enumerate deleted top-level packages;
  unlisted deletions fail closed.

### Lesson 2: Trust machinery must be adversarially tested before its first real vote

Evidence from this session:

- irc-conv-v1 was APPROVED by a tally that ignores quorum= (B2). A one-line
  test (TestTallyHonorsQuorumAll) exposes it; that test did not exist.

Protocol implication:

- A tally/merge-gate change is itself shared-state: it needs its own ccrep
  round with failing-then-passing test evidence before being relied upon.

Code/tooling implication:

- Quorum semantics need a definition in code: quorum=all over present
  principals, presence derived from JOIN/QUIT in the same ledger.

### Lesson 3: Message-loss incidents cluster around client restarts, not network failures

Evidence from this session:

- All three loss events (512-byte crash 10:18, reconnect window 10:21, agy
  offline 10:24) were client-lifecycle gaps; ngircd itself never dropped.

Protocol implication:

- Replay-on-join must be mandatory before acting, and the substrate should
  provide it natively (ergo CHATHISTORY) rather than by convention.

Code/tooling implication:

- irc-client should fetch/read backlog automatically on connect before exposing
  the FIFO.

### Lesson 4: Worktree-relative paths cannot express shared state

Evidence from this session:

- B1: scribe wrote doc/collab/history.jsonl in wt-agy (200 lines); the same
  relative path in wt-claude and the main checkout was empty/absent.

Protocol implication:

- Every shared artifact needs a fam-root-derived absolute location, outside all
  worktrees, declared in PROTOCOL.md.

Code/tooling implication:

- COLLAB_HISTORY env + default of <fam-root>/collab/history.jsonl; refuse to
  start the scribe with a repo-relative path.

### Lesson 5: Buy the substrate, build the domain logic

Evidence from this session:

- pirc (300-line toy) cost two debugging rounds (IPv6, ircII handshake); ngircd
  worked immediately. The 160-line scribe/tally, by contrast, is
  domain-specific and no off-the-shelf tool replaces it.

Protocol implication:

- Substrate choices (server, history, auth) should default to battle-tested
  components; protocol-vocabulary tools are the build surface.

## 7. Things That Went Poorly

### Went poorly: agy's cutover bundled ratified steps 3+4

Evidence:

- 6fc3930 deletes internal/server, internal/store/sqlite.go, voting.go and 1707
  integration-test lines in the same commit that adds the IRC stack.

Impact:

- No fallback path mid-cutover; review burden concentrated in one 11k-line
  diff; blocked landing.

Likely system cause:

- No machine link between ratified proposal text and permissible diff scope.

Could automation prevent this?

- yes (Lesson 1 merge-gate check).

Suggested fix:

- Split deletion into its own proposal post-cutover; add diff-scope gate.

### Went poorly: claude's wake mechanism failed silently twice over

Evidence:

- tail -f | grep -m1 pipeline matched at 10:09:48 but never completed (SIGPIPE
  deferred); two later watcher launches used `&` instead of a tracked
  background task and could never notify.

Impact:

- Missed a live operator request (date check); risked indefinite deafness.

Likely system cause:

- Ad-hoc shell patterns for a load-bearing capability; no canonical "wait for
  channel event" verb in the tooling.

Could automation prevent this?

- yes.

Suggested fix:

- Add `botfam irc-wait [--from ...] [--pattern ...]` that blocks until a
  matching ledger line, replacing per-agent shell watchers.

### Went poorly: misdiagnosis of the first tally failure

Evidence:

- 10:32:56 claude asserted the scribe lacked replay-on-start; actual cause was
  a stale scribe binary from an earlier background task (agy, 10:33:39).

Impact:

- Minor (corrected in one exchange), but the prescribed fix was wrong.

Likely system cause:

- No way to ask a bot its version/identity; two scribe processes were
  indistinguishable on the channel.

Suggested fix:

- Bots answer !version with binary build info; stable nicks (already adopted)
  plus ghost-on-reclaim prevent dual-process ambiguity.
