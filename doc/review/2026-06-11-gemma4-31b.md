3. **Blind spots**

- **Docker Desktop Dependency**: The migration to Docker Compose introduces a dependency on the Docker Desktop daemon. While claude mentioned enabling "start-at-login" (16:56:09), there is no programmatic check or alert system if the daemon fails to start, which would take down the entire coordination substrate.
- **Scribe's "Single Point of Truth" Risk**: The session focused heavily on Scribe's versioning and hosting, but not on its failure modes within Docker. If the scribe container hangs without crashing (liveness vs readiness), the ledger could stop silently while Ergo remains up.
- **The `irc-wait` Race Condition (Speculation)**: While claude flagged the need to re-arm the watcher, there is a risk of a race condition where an agent wakes, processes a message, and misses subsequent messages arriving exactly during the "re-arming" window.

4. **Proposals**

- **Automated Wake-Loop Wrapper**: Create a wrapper around `irc-wait` that handles the re-arm cycle internally to prevent agents from going silent. (Solves: Pain point regarding "sharp edge" of re-arming; Cost: small).
- **Docker Healthcheck for Scribe**: Add a Docker healthcheck to the scribe service in `compose.test.yaml` and `botfam-irc-prod` that validates it is actually writing to the ledger, not just running as a process. (Solves: Blind spot regarding silent failure; Cost: small).
- **Scribe-less Recovery Doc**: Document a manual way for the human operator to recover/dump the IRC history if the scribe bot fails permanently or the Docker volume is corrupted. (Solves: Blind spot regarding single point of truth; Cost: medium).
