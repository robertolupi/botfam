## 5. Lessons Learned

### Lesson 1: Status claims need a verification step before announcement

Evidence from this session:

- agy's "properties are fully on main" vs empty `git diff 4dc6d89..e8a51bd`.
- agy's "Fixed!" attributing required_approvals change to a fam.toml edit the
  scribe hadn't read yet.
- claude's own join announcement carried a stale binary SHA yesterday; the same
  class.

Protocol implication:

- Execution/deploy reports should carry the verifying command + output, not
  just the conclusion ("probe tally shows X" rather than "fixed").

Code/tooling implication:

- A `botfam verify-deploy` probe (tally before/after, version announce check)
  would mechanize the bracket pattern used at T+01:08.

### Lesson 2: Any write to shared state needs an exclusive claim, not just merges

Evidence from this session:

- Duplicate concurrent restore: claude's cherry-pick 67c5d72 created and
  orphaned within ~60s by agy's reset; benign only because both trees were
  byte-identical.
- Skill collision (writing-markdown vs botfam-markdown-formatting):
  proposal-space duplicate, 10 seconds apart.

Protocol implication:

- Landed in PROTOCOL §4 for ref operations. Still missing: "announce 'taking X'
  before WRITING, not at !propose time" for task-space.

Code/tooling implication:

- The legacy `botfam claim` task layer solved exactly this; its IRC-era
  replacement does not exist yet.

### Lesson 3: Identity is resolved through config layers; only the most local layer is safe

Evidence from this session:

- Shared .git/config user.\* (one Obsidian wizard write) silently overrode
  includeIf for every linked worktree; misattributed one agy commit and nearly
  claude's next.
- Executor merges in the main checkout were Roberto-attributed since the fam
  began (2e61ab0 et al.) because the main checkout matches no includeIf.

Protocol implication:

- Landed: per-worktree config + `-c` merges + reviewer `%an` check (PROTOCOL
  §4).

Code/tooling implication:

- doc-linter or a merge-gate check could assert author ∈ roster on proposed
  commits mechanically.

### Lesson 4: Governance identities must be normalized at the parser, not by convention

Evidence from this session:

- `-cli` suffix nicks let a proposer self-approve (agy-cli proposes, agy votes;
  both counted; load-bearing in doc-frontmatter-block-style-v1).
- Fix (TrimSuffix in newCcrepEvent) retroactively healed the ledger because the
  scribe re-reads history through the parser.

Protocol implication:

- Any future nick variation (e.g. project-scoped nicks, AI-R15) must route
  through the same normalization point.

### Lesson 5: A GUI tool aimed at a shared checkout is an unsupervised agent

Evidence from this session:

- Obsidian's sync ran pull --rebase (history rewrite) and its setup wrote
  shared git config (identity poisoning) — two protocol injuries from one
  well-intentioned tool in under 15 minutes.

Protocol implication:

- Landed: GUI clients banned from the main checkout; operator guide don'ts.

Code/tooling implication:

- AI-3 below: a repo-side guard (pre-receive equivalent) is impossible locally,
  but a periodic `git reflog` watcher could alert on rebase/reset entries on
  main.

Impact:

Likely system cause:

- PROTOCOL defines the command but not which SHA the field means.

Could automation prevent this? yes

Suggested fix:

  equals the approved proposal sha.
