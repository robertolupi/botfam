## Failure & Edge Case Analysis

### 1. The Worktree Version-Reporting Trap

During the roll-out of build stamping, the initial version fallback executed a shell command (`git rev-parse HEAD`) when compiled binaries were executed.

* **The Vulnerability:** If a stale global binary (`~/bin/botfam`) was executed inside an un-synced or dirty git worktree, it would report the state of the host directory rather than its actual compiled state.
* **The Fix:** Code was immediately modified to extract immutable compiler-level metadata directly using Go’s runtime `ReadBuildInfo`, utilizing `vcs.modified` to explicitly flag `-dirty` builds.

### 2. Scribe Double-Append Vulnerability

Migrating from `launchd` to Docker meant changing the execution harness of `scribe`. Had the host-level `launchd` job not been gracefully and verified as unloaded (`launchctl bootout`) prior to the Docker container spin-up, two separate instances of `scribe` would have sat on the same IRC channel, logging twin copies of every event and corrupting the canonical database. The strict execution gate handled by the human operator (`rlupi` issuing a manual ping confirmation check) averted this state collision.
